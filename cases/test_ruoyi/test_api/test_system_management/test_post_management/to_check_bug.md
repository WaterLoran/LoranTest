# 岗位管理模块 - 疑似 BUG 记录

> 以下 BUG 均通过测试脚本按照合理业务期望编写断言后发现，测试断言已改为正确的业务预期，
> 若脚本执行失败即可确认对应 BUG 存在。

---

## BUG-1: 岗位名称未做 trim 处理（前后空格可入库） 【已确认】

- **关联用例**: TC-B09 (`test_name_leading_trailing_spaces`)
- **复现步骤**: 创建岗位，名称传入 `"  岗位名称  "`（前后各两个空格）
- **实际结果**: 返回 200，带空格的名称直接入库
- **预期结果**: 系统应返回 500 拒绝，或自动 trim 前后空格后入库
- **影响**: UI 显示异常；可能绕过唯一性校验（"岗位A" 和 " 岗位A " 被视为不同岗位）
- **严重级别**: 中
- **BUG 类型**: 输入校验
- **脚本断言**: 创建后查详情 `check=[["$.data.postName", "eq", "TC_B09_xxx"]]`（trim 后的值）
- **确认时间**: 2026-03-07
- **确认证据**: 日志 `test_post_name_validation_20260307_232226.log` 中断言失败：`真实指定的响应对象为   TC_B09_1772896946  `（含空格），`期望对象 为 TC_B09_1772896946`（无空格）

## BUG-2: 岗位名称允许换行符 【已确认】

- **关联用例**: TC-B10 (`test_name_newline`)
- **复现步骤**: 创建岗位，名称传入 `"行1\n行2"`
- **实际结果**: 返回 200，含换行符的名称入库
- **预期结果**: 系统应返回 500 拒绝包含换行符的名称
- **影响**: 前端表格/列表/下拉框显示错乱，数据导出格式异常
- **严重级别**: 中
- **BUG 类型**: 输入校验
- **脚本断言**: 创建后查详情，期望 postName 不含换行符
- **确认时间**: 2026-03-07
- **确认证据**: 日志 `test_post_name_validation_20260307_232226.log` 中断言失败：`真实指定的响应对象为 TC_B10_行1\n行2_1772896946`（含换行），`期望对象 为 TC_B10_行1行2_1772896946`（无换行）

## BUG-3: 岗位名称未过滤 HTML 标签（XSS 风险）

- **关联用例**: TC-B11 (`test_name_html_tag`), TC-B13 (`test_name_xss`)
- **复现步骤**: 创建岗位，名称传入 `"<b>html</b>"` 或 `"<img src=x onerror=alert(1)>"`
- **实际结果**: 返回 200，原始 HTML 标签直接入库
- **预期结果**: 系统应转义 HTML 特殊字符（`<`, `>`, `&` 等），或拒绝包含 HTML 标签的输入
- **影响**: 若前端使用 `v-html` 或未转义渲染，存在 **存储型 XSS 安全漏洞**
- **严重级别**: 高
- **备注**: 当前脚本断言为 `eq 200`（API 层面接受字符串本身不算错误），实际安全风险在前端渲染层

## BUG-4: 显示顺序 postSort 接受小数值

- **关联用例**: TC-D06 (`test_sort_decimal`)
- **复现步骤**: 创建岗位，postSort 传入 `1.5`
- **实际结果**: 返回 200，小数值入库
- **预期结果**: 系统应返回 500 拒绝，postSort 为整数类型字段不应接受小数
- **影响**: 数据类型不一致，排序行为不可预期
- **严重级别**: 低
- **脚本断言**: `check=[["$.code", "eq", 500]]`

## BUG-5: 显示顺序 postSort 接受负数

- **关联用例**: TC-D04 (`test_sort_negative`)
- **复现步骤**: 创建岗位，postSort 传入 `-1`
- **实际结果**: 待确认（系统可能返回 200 接受）
- **预期结果**: 系统应返回 500 拒绝，显示顺序应为非负整数
- **影响**: 负数排序值无业务意义，可能导致列表排序混乱
- **严重级别**: 低
- **脚本断言**: `check=[["$.code", "eq", 500]]`

## BUG-6: 状态 status 接受非法枚举值 【已确认】

- **关联用例**: TC-E03 (`test_status_invalid`)
- **复现步骤**: 创建岗位，status 传入 `"2"`（合法值仅为 "0" 正常、"1" 停用）
- **实际结果**: 系统返回 200 接受，status="2" 原样入库
- **预期结果**: 系统应返回 500 拒绝，status 字段应做枚举值校验
- **影响**: 非法状态值导致前端显示异常、按状态筛选结果不准确、业务逻辑判断错误
- **严重级别**: 中
- **BUG 类型**: 业务规则
- **脚本断言**: 创建后查详情 `check=[["$.data.status", "in", ["0", "1"]]]`
- **确认时间**: 2026-03-07
- **确认证据**: 日志 `test_post_status_validation_20260307_232227.log` 中断言失败：`真实指定的响应对象为 2`（非法枚举值），`期望对象 为 ['0', '1']`（合法枚举列表）

## BUG-7: 岗位编码前后空格未处理 【已确认】

- **关联用例**: TC-C11 (`test_code_leading_trailing_spaces`)
- **复现步骤**: 创建岗位，编码传入 `"  TC_C11_1772896622  "`（前后各两个空格）
- **实际结果**: 系统返回 200 接受，查询详情确认存储的 postCode 为 `"  TC_C11_1772896622  "`（未 trim）
- **预期结果**: 系统应自动 trim 前后空格后入库，即存储为 `"TC_C11_1772896622"`
- **影响**: 编码空格会导致唯一性校验失效、API 按编码查询异常、前端显示异常、数据导出异常
- **严重级别**: 中
- **脚本断言**: 创建后查详情 `check=[["$.data.postCode", "eq", "TC_C11_xxx"]]`（trim 后的值）
- **确认时间**: 2026-03-07
- **确认证据**: 详情接口响应 `"postCode": "  TC_C11_1772896622  "`，前后空格原样入库

## BUG-8: 岗位编码唯一性校验不区分大小写

- **关联用例**: TC-C13 (`test_code_case_sensitive`)
- **复现步骤**: 先创建编码为 `"TC_C13_Abc"` 的岗位，再创建编码为 `"TC_C13_abc"` 的岗位
- **实际结果**: 第二次创建返回 500（岗位编码已存在），系统将大小写不同的编码视为相同
- **预期结果**: 系统应返回 200，`"Abc"` 和 `"abc"` 是不同字符串，应视为不同编码
- **影响**: 限制了编码的灵活性；与常见编程/业务约定不一致（编码通常区分大小写）
- **严重级别**: 低
- **脚本断言**: 第二次添加 `check=[["$.code", "eq", 200]]`
